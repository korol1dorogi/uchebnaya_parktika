## 0. Что вообще мы построили (обзор проекта)

Веб-сервис, генерирующий позитивные хокку, с регистрацией/входом и хранением
в БД. Состоит из **трёх контейнеров** в одной сети:

```
            ┌─────────────┐   POST /hash, /verify   ┌──────────────┐
 браузер ─► │  praktika   │ ──────────────────────► │   hasher     │
  HTTP      │  (Drogon,   │                         │ (Python/Flask│
            │   C++20)    │ ──┐                      │  хэш паролей)│
            └─────────────┘   │ SQL                  └──────────────┘
                              ▼
                        ┌──────────────┐
                        │  PostgreSQL  │
                        └──────────────┘
```

- **praktika** — основной HTTP-сервис на C++ (фреймворк Drogon). Эндпоинты API
  и HTML-страницы. Слушает порт 8081 (наружу пробрасываем на 8082).
- **hasher** — отдельный Python/Flask микросервис, который хэширует и проверяет
  пароли (praktika сам этого не делает).
- **db** — PostgreSQL: таблицы пользователей и хокку.

Поверх этого: запуск в **Docker Compose** и в **Kubernetes (minikube)**,
**юнит/интеграционные тесты** и **CI/CD на GitHub Actions**.

## 1. Клиент-серверная архитектура и REST API

### Теория
- **Клиент-сервер** — модель, где **клиент** (браузер, мобильное приложение,
  curl) посылает **запрос**, а **сервер** обрабатывает и шлёт **ответ**.
  Логика и данные — на сервере (бэкенд), клиент только отображает/инициирует.
- **HTTP** — протокол запрос-ответ. Запрос содержит: **метод**, **путь (URL)**,
  заголовки, тело. Ответ: **статус-код**, заголовки, тело.
- **Методы HTTP** (глаголы):
  - `GET` — получить данные (не меняет состояние);
  - `POST` — создать/отправить;
  - `PUT`/`PATCH` — обновить;
  - `DELETE` — удалить.
- **Статус-коды**:
  - `2xx` — успех (`200 OK`, `201 Created`);
  - `4xx` — ошибка клиента (`400` неверный запрос, `401` не авторизован,
    `404` не найдено, `409` конфликт);
  - `5xx` — ошибка сервера (`500`, `502` плохой ответ от внешнего сервиса).
- **API** — программный интерфейс: набор эндпоинтов, через которые клиент
  работает с сервером.
- **REST** — стиль построения API:
  - ресурсы адресуются URL (`/api/haikus/{id}`),
  - действие задаётся HTTP-методом,
  - **stateless** — сервер не хранит состояние клиента между запросами в
    самом протоколе (состояние — в БД/сессии),
  - обмен обычно в **JSON**.

### Практика (наши эндпоинты)
| Метод | Путь | Действие |
|---|---|---|
| GET | `/api/hello` | вернуть `Hello, World!` |
| GET | `/api/happy` | хокку от Groq + сохранение в БД |
| POST | `/api/register` | регистрация |
| POST | `/api/login` / `/api/logout` | вход / выход |
| GET | `/api/haikus`, `/api/haikus/my` | список всех / своих хокку |
| PUT | `/api/haikus/{id}/comment` | прокомментировать |
| DELETE | `/api/haikus/{id}` | удалить |

### Вероятные вопросы
- *Что такое REST?* — Архитектурный стиль: ресурсы по URL, действия по
  HTTP-методам, без состояния, обычно JSON.
- *Чем GET отличается от POST?* — GET читает (идемпотентен, без тела), POST
  создаёт/изменяет (с телом).
- *Что значит «stateless»?* — Каждый запрос самодостаточен; сервер не помнит
  предыдущий. Идентификация — через токен/cookie сессии в каждом запросе.

## 2. Docker

### Теория
- **Docker** — платформа **контейнеризации**. **Контейнер** — изолированный
  процесс с собственной файловой системой и зависимостями.
- **Контейнер vs виртуальная машина:** ВМ тащит целую гостевую ОС с ядром,
  контейнер использует **ядро хоста** → он легче, быстрее стартует, меньше
  весит.
- **Образ (image)** — неизменяемый шаблон (файловая система + метаданные).
  **Контейнер** — запущенный экземпляр образа (их можно много из одного образа).
- **Dockerfile** — рецепт сборки образа.
- **Слои (layers):** каждая инструкция Dockerfile создаёт слой; слои
  **кэшируются**. Поэтому неизменившиеся слои не пересобираются → быстрая
  пересборка. (Отсюда правило: сначала копировать то, что меняется редко —
  зависимости, потом код.)
- **Реестр (registry):** хранилище образов (Docker Hub, GHCR). `pull`/`push`.
- **Volume (том):** механизм постоянного хранения; данные переживают
  удаление контейнера (контейнеры эфемерны, тома — нет).
- **Сеть:** контейнеры можно объединять в сети, где они видят друг друга.
- **Multi-stage build:** сборка в несколько стадий — в одной собираем
  (тяжёлый тулчейн), в финальную кладём только результат → маленький образ.

### Основные инструкции Dockerfile
| Инструкция | Что делает |
|---|---|
| `FROM` | базовый образ |
| `WORKDIR` | рабочая директория |
| `COPY` / `ADD` | копировать файлы в образ |
| `RUN` | выполнить команду при сборке (создаёт слой) |
| `ENV` | переменная окружения |
| `EXPOSE` | задокументировать порт |
| `CMD` | команда по умолчанию при запуске контейнера |
| `ENTRYPOINT` | «несменяемая» точка входа |

**`CMD` vs `ENTRYPOINT`:** `ENTRYPOINT` — что запускать всегда; `CMD` — аргументы
по умолчанию (их легко переопределить в `docker run`).

### Основные команды
| Команда | Что делает |
|---|---|
| `docker build -t имя:тег .` | собрать образ |
| `docker images` | список образов |
| `docker run -d -p 8082:8081 имя` | запустить (фоном, проброс порта хост:контейнер) |
| `docker ps` / `-a` | запущенные / все контейнеры |
| `docker logs -f <c>` | логи |
| `docker exec -it <c> sh` | shell внутри |
| `docker stop/start/rm <c>` | остановить/запустить/удалить контейнер |
| `docker rmi <img>` | удалить образ |
| `docker pull/push <img>` | скачать/выгрузить из реестра |
| `docker volume ls` | список томов |
| `docker system prune -a` | очистить мусор |

### Практика: наш Dockerfile (multi-stage)
```dockerfile
FROM drogonframework/drogon AS build      # стадия сборки (есть Drogon+тулчейн)
WORKDIR /src
COPY . .
RUN cmake -S . -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build -j

FROM drogonframework/drogon               # финальная стадия
WORKDIR /app
COPY --from=build /src/build/praktika ./  # берём только бинарник
COPY config.json ./
EXPOSE 8081
CMD ["./praktika"]
```
Сборка/запуск: `docker build -t praktika . && docker run -p 8082:8081 praktika`.

### Вероятные вопросы
- *Чем образ отличается от контейнера?* — Образ — шаблон (неизменяемый),
  контейнер — запущенный экземпляр образа.
- *Что такое слой и зачем он?* — Результат одной инструкции Dockerfile;
  кэшируется, ускоряет пересборку.
- *Зачем multi-stage?* — Чтобы в финальный образ не тащить компилятор и
  исходники — только готовый бинарник. Меньше размер и поверхность атаки.
- *Как сохранить данные между перезапусками контейнера?* — Через **volume**.
- *Контейнер vs ВМ?* — Контейнер делит ядро хоста, легче и быстрее ВМ.
- *Как пробросить порт?* — `-p хостовый:контейнерный`.

## 3. Docker Compose

### Теория
- **Compose** описывает **многоконтейнерное** приложение в одном
  `docker-compose.yml` и поднимает его одной командой.
- **service** — один контейнер с настройками (образ/сборка, порты, env, тома,
  зависимости).
- **Сеть:** Compose создаёт общую сеть; сервисы обращаются друг к другу **по
  имени сервиса** (DNS). У нас praktika ходит к `db` и `hasher` по этим именам.
- **`.env`:** Compose автоматически читает файл `.env` и подставляет
  `${ПЕРЕМЕННАЯ}` — так держим **секреты вне кода**.
- **`depends_on` / `healthcheck`:** порядок запуска и проверка готовности.

### Основные команды
| Команда | Что делает |
|---|---|
| `docker compose up -d --build` | собрать и поднять весь стек в фоне |
| `docker compose down` | остановить и удалить контейнеры/сеть |
| `docker compose down -v` | то же + удалить тома (данные БД!) |
| `docker compose ps` | статус сервисов |
| `docker compose logs -f [svc]` | логи всех/сервиса |
| `docker compose build [svc]` | пересобрать |
| `docker compose restart [svc]` | перезапустить |
| `docker compose exec <svc> sh` | shell в сервисе |

### Практика: фрагмент нашего compose
```yaml
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}     # из .env
    volumes:
      - db_data:/var/lib/postgresql/data    # постоянные данные
  hasher:
    build: ./hasher
  praktika:
    build: .
    depends_on: [db, hasher]
    ports: ["8082:8081"]
    environment:
      - DB_HOST=db                          # обращение к сервису по имени
      - HASHER_URL=http://hasher:5005
volumes:
  db_data:
```
Запуск: `cp .env.example .env` → `docker compose up -d --build`.

### Вероятные вопросы
- *Зачем Compose, если есть docker run?* — Чтобы поднять **много связанных**
  контейнеров одной командой, с сетью/томами/env, декларативно.
- *Как сервисы находят друг друга?* — По **имени сервиса** через встроенный DNS.
- *Где хранятся данные БД?* — В именованном **volume** (`db_data`); `down -v`
  его удалит.
- *Чем `down` отличается от `down -v`?* — `-v` дополнительно удаляет тома.

## 4. REST API на C++ (фреймворк Drogon)

### Теория
- **Drogon** — асинхронный C++ веб-фреймворк (HTTP-сервер + роутинг + ORM).
  Работает на **event loop** (неблокирующий I/O) — один поток обслуживает много
  соединений.
- **Контроллер** — класс с методами-обработчиками. Наследуем
  `drogon::HttpController<T>`; методы **сами регистрируются** на маршруты через
  макрос `ADD_METHOD_TO`. Drogon находит контроллеры из папки `controllers/`
  автоматически (в CMake — `aux_source_directory`).
- **Корутины (C++20):** вместо «лапши» из колбэков пишем последовательный код
  с `co_await`. `co_await db->execSqlCoro(...)` — асинхронный запрос к БД, но
  выглядит как обычный вызов.
- **JSON:** Drogon строит ответы через `Json::Value` и
  `HttpResponse::newHttpJsonResponse`.

### Практика: как сделать новый эндпоинт
1. Создать `controllers/MyController.h`:
```cpp
class MyController : public drogon::HttpController<MyController> {
 public:
  METHOD_LIST_BEGIN
  ADD_METHOD_TO(MyController::hello, "/api/hello", Get);   // путь + метод
  METHOD_LIST_END
  void hello(const HttpRequestPtr& req,
             std::function<void(const HttpResponsePtr&)>&& cb) const;
};
```
2. Реализовать в `.cc`:
```cpp
void MyController::hello(const HttpRequestPtr&,
        std::function<void(const HttpResponsePtr&)>&& cb) const {
  auto resp = HttpResponse::newHttpResponse();
  resp->setBody("Hello, World!");
  cb(resp);
}
```
3. Пересобрать — файл подхватится автоматически.

Путь-параметры: `ADD_METHOD_TO(C::m, "/api/haikus/{1}", Delete);` → метод
получает доп. аргумент `std::string id`.

### Вероятные вопросы
- *Что такое асинхронность/event loop?* — Сервер не блокируется на I/O: пока
  ждёт БД/сеть, обрабатывает другие запросы. Один поток — много соединений.
- *Что такое корутина и зачем `co_await`?* — Способ писать асинхронный код
  последовательно (без вложенных колбэков); `co_await` «приостанавливает»
  функцию до готовности результата, не блокируя поток.
- *Как добавить роут?* — Контроллер + `ADD_METHOD_TO(метод, путь, HTTP-метод)`.

## 5. Базы данных и PostgreSQL

### Теория
- **Реляционная БД** хранит данные в **таблицах** (строки = записи, столбцы =
  поля). PostgreSQL — реляционная СУБД.
- **Ключи:**
  - **PRIMARY KEY (PK)** — уникальный идентификатор строки;
  - **FOREIGN KEY (FK)** — ссылка на PK другой таблицы (связь);
  - **UNIQUE** — значение не повторяется (у нас `login`).
- **NULL** — «нет значения» (у нас `user_uuid` хокку = NULL означает аноним).
- **Индекс** — структура для быстрого поиска по столбцу (ускоряет `WHERE`,
  замедляет вставку).
- **SQL** — язык запросов:
  - `SELECT ... FROM ... WHERE ...` — чтение;
  - `INSERT INTO ... VALUES ...` — вставка;
  - `UPDATE ... SET ... WHERE ...` — обновление;
  - `DELETE FROM ... WHERE ...` — удаление;
  - `JOIN` — соединение таблиц.
- **Транзакция** — группа операций «всё или ничего» (ACID).

### Практика: наша схема (`db/init.sql`)
```sql
CREATE TABLE users (
  uuid     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  login    TEXT NOT NULL UNIQUE,
  password TEXT NOT NULL,      -- строка salt:iterations:hash
  salt     TEXT NOT NULL
);
CREATE TABLE haikus (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  text       TEXT NOT NULL,
  user_uuid  UUID NULL REFERENCES users(uuid) ON DELETE SET NULL,  -- FK, NULL=аноним
  comment    TEXT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### Практика: подключение из C++ (Drogon) и CRUD
- Клиент БД создаётся в `main.cc` из переменных окружения (пароль **не** в
  коде):
```cpp
drogon::app().createDbClient("postgresql", host, port, db, user, password, 1);
```
- Запрос с **параметрами** (`$1`, `$2` — защита от SQL-инъекций):
```cpp
auto db = app().getDbClient();
auto r = co_await db->execSqlCoro(
    "SELECT uuid FROM users WHERE login=$1", login);
if (!r.empty()) std::string id = r[0]["uuid"].as<std::string>();
```
- Вставка с возвратом id: `... INSERT ... RETURNING id`.
- Проверка изменённых строк: `r.affectedRows()`. Проверка NULL: `field.isNull()`.

### Контейнеризация БД
В Compose — сервис `postgres:16-alpine` + том для данных + монтирование
`db/init.sql` в `/docker-entrypoint-initdb.d` (выполняется при первом старте).

### Вероятные вопросы
- *Что такое PRIMARY KEY / FOREIGN KEY?* — PK уникально идентифицирует строку;
  FK ссылается на PK другой таблицы (связь между таблицами).
- *Что такое индекс?* — Структура для быстрого поиска по столбцу.
- *Что такое SQL-инъекция и как защищаемся?* — Подмена SQL через ввод
  пользователя. Защита — **параметризованные запросы** (`$1`, значения
  передаются отдельно, не склеиваются в строку).
- *Что такое транзакция?* — Набор операций «всё или ничего» (ACID).
- *Как реализован CRUD?* — Create=`INSERT`, Read=`SELECT`, Update=`UPDATE`,
  Delete=`DELETE`, через `execSqlCoro`.

## 6. Архитектура сервиса: микросервис, авторизация, хэширование

### Теория
- **Микросервис** — отдельный маленький сервис с одной задачей. У нас `hasher`
  отвечает только за хэширование паролей. Плюсы: изоляция, можно писать на
  другом языке (Python), масштабировать отдельно.
- **Хэширование паролей:** пароль нельзя хранить в открытом виде. Храним
  **хэш** — необратимое преобразование. **Соль (salt)** — случайные байты,
  добавляемые к паролю, чтобы одинаковые пароли давали разные хэши (защита от
  радужных таблиц). **Итерации** — много раундов, чтобы подбор был медленным.
  Формат у нас: `salt:iterations:hash`.
  > В проекте `simple_kdf` **учебная, не криптостойкая** (для реального прода —
  > argon2/scrypt/bcrypt).
- **Хэш vs шифрование:** хэш **необратим** (нельзя «расшифровать»), шифрование
  обратимо при наличии ключа.
- **Сессии и cookie:** после входа сервер заводит **сессию** и отдаёт браузеру
  **cookie** (`SESSION`) с идентификатором. Браузер шлёт cookie в каждом
  запросе → сервер узнаёт пользователя. У нас сессии Drogon (in-memory).

### Практика
- Регистрация: praktika → `POST /hash` (hasher) → сохраняет в БД → ставит сессию.
- Вход: praktika → `POST /verify` (hasher) → при успехе ставит сессию.
- Эндпоинты hasher: `POST /hash` (вернуть `salt:iters:hash`), `POST /verify`.

### Вероятные вопросы
- *Зачем хэшировать пароль, а не хранить как есть?* — Чтобы при утечке БД
  пароли нельзя было прочитать.
- *Зачем соль?* — Чтобы одинаковые пароли давали разные хэши; ломает радужные
  таблицы.
- *Чем хэш отличается от шифрования?* — Хэш необратим, шифрование обратимо
  ключом.
- *Что такое сессия и cookie?* — Сессия — состояние «кто вошёл» на сервере;
  cookie — идентификатор сессии у клиента, шлётся в каждом запросе.
- *Зачем выносить хэширование в отдельный сервис?* — Изоляция
  ответственности, независимое развитие/масштабирование, другой язык.

## 7. Kubernetes (кубер)

### Теория
- **Kubernetes (k8s)** — **оркестратор контейнеров**. Ты декларативно
  описываешь **желаемое состояние** (YAML), а кластер через **контроллеры
  (control loop)** постоянно приводит реальность к нему.
- Что даёт сверх Compose: **self-healing** (упал под → поднимет), **масштаб**,
  **rolling-update** (обновление без простоя), **service discovery** и
  балансировку, работу на **многих узлах**.
- **minikube** — локальный кластер из одного узла для учёбы. **kubectl** —
  утилита управления кластером.

### Объекты
| Объект | Назначение |
|---|---|
| **Pod** | минимальная единица: 1+ контейнеров с общей сетью |
| **ReplicaSet** | держит заданное число одинаковых подов |
| **Deployment** | управляет ReplicaSet: реплики, обновления, self-healing |
| **Service** | стабильное имя/IP + балансировка на поды |
| **ConfigMap** | несекретная конфигурация (как env) |
| **Secret** | секреты (как env, отдельно от кода) |
| **PersistentVolume / PVC** | постоянное хранилище (для БД) |
| **Namespace** | изоляция группы ресурсов |
| **probes** | readiness (готов к трафику) / liveness (жив, иначе рестарт) |
| **HPA** | автоскейл по нагрузке |

- **Цепочка:** `Deployment → ReplicaSet → Pod`. Удалил под — ReplicaSet
  создаёт новый (**self-healing**).
- **Service, типы:** **ClusterIP** (виден только внутри — у нас `db`, `hasher`),
  **NodePort** (проброшен наружу — у нас `praktika`).
- **probes:** *readiness* — пока не готов, Service не шлёт трафик;
  *liveness* — если не отвечает, контейнер перезапускается.
- **Поды эфемерны** → данные хранят в **PVC** (у Postgres).

### Образы в minikube
У minikube **свой Docker-демон** — локальные образы он не видит. Поэтому:
`minikube image load praktika:latest` + в манифесте `imagePullPolicy: IfNotPresent`.

### Практика: автоскейл (HPA) — рецепт
HPA сам меняет число реплик по нагрузке CPU. Нужны: **metrics-server** и
**resource requests** в Deployment.
```bash
minikube addons enable metrics-server
kubectl apply -f k8s/hpa.yaml          # min 1, max 5, цель CPU 50%
# нагрузка:
kubectl -n praktika run loadgen --image=busybox --restart=Never -- /bin/sh -c \
  'for i in $(seq 1 60); do (while true; do wget -q -O /dev/null http://praktika:8081/api/hello; done) & done; sleep 600'
kubectl -n praktika get hpa,pods -w    # реплики растут, после снятия — падают
```
**Как считает:** `желаемые_реплики = ceil(текущие × текущий_CPU / целевой_CPU)`.
Например, при 118% и цели 50%: `ceil(1×118/50)=3`.

### Вероятные вопросы
- *Чем k8s лучше Compose?* — Self-healing, масштаб, обновления без простоя,
  много узлов, service discovery. Compose — для одной машины/дева.
- *Чем Deployment отличается от Pod?* — Pod — единичный экземпляр; Deployment
  управляет множеством подов (реплики, восстановление, обновления).
- *Зачем Service?* — Стабильный адрес и балансировка: поды эфемерны и меняют
  IP, а Service — постоянный вход к ним.
- *ClusterIP vs NodePort?* — ClusterIP виден только внутри кластера, NodePort
  пробрасывает порт наружу.
- *readiness vs liveness?* — readiness: готов ли принимать трафик; liveness:
  жив ли (иначе рестарт).
- *Как работает self-healing?* — ReplicaSet следит за числом реплик; не хватает
  — создаёт новые поды.
- *Что нужно для HPA?* — metrics-server + resource requests (CPU) + манифест HPA.
- *Почему данные БД в PVC, а не в поде?* — Поды эфемерны; PVC хранит данные
  отдельно и переживает пересоздание пода.

## 8. Тестирование

### Теория
- **Пирамида тестов:** много быстрых **юнит**-тестов (отдельные функции),
  меньше **интеграционных** (несколько модулей вместе), ещё меньше **e2e**.
- **Юнит-тест** проверяет **одну единицу** изолированно (чистую функцию).
  Детерминированно и быстро.
- **Интеграционный/smoke** проверяет, что компоненты **работают вместе**
  (реальная сеть/БД).
- **Мок (mock)** — подмена зависимости заглушкой (мы не используем, но знать).

### Практика
- **Python (hasher) — pytest** (`hasher/tests/test_app.py`): чистые функции
  (`simple_kdf`, hash/verify) + эндпоинты через **`app.test_client()`** (фейковый
  HTTP-клиент без реального сервера). Запуск: `pytest hasher/tests`.
- **C++ — GoogleTest** (`tests/`): отдельный проект (через **FetchContent**
  скачивает GoogleTest), тестирует чистые утилиты `utils/`. Макрос `TEST(...)`,
  проверки `EXPECT_EQ/EXPECT_TRUE`. Запуск через **CTest**:
  `cmake -S tests -B build-tests && ctest --test-dir build-tests`.
- **Интеграционный smoke:** поднять стек `docker compose up` и дёрнуть
  `/api/hello`, register, login (проверяет связку praktika+hasher+db).

### Вероятные вопросы
- *Чем юнит отличается от интеграционного?* — Юнит — одна функция изолированно;
  интеграционный — несколько компонентов вместе.
- *Что такое мок?* — Заглушка вместо реальной зависимости.
- *Зачем тесты?* — Ловить регрессии автоматически, а не руками.
- *Что такое test_client?* — Встроенный фейковый HTTP-клиент Flask: дёргает
  приложение в процессе, без сети.

## 9. CI/CD (GitHub Actions)

### Теория
- **CI (Continuous Integration)** — на каждый push/PR автоматически собирать и
  **прогонять тесты**, чтобы поломка ловилась сразу.
- **CD (Continuous Delivery/Deployment)** — автоматически доставлять/деплоить
  (пуш образа в реестр, выкат в кластер). Мы сделали **CI**.
- **GitHub Actions** — встроенный в GitHub CI. Конфиг — YAML в
  `.github/workflows/`.
  - **`on`** — триггеры (push, pull_request).
  - **`jobs`** — независимые задачи; бегут **параллельно**, каждая на чистой
    одноразовой ВМ (`runs-on`).
  - **`steps`** — шаги: `uses:` (готовый экшен, напр. `actions/checkout`) или
    `run:` (команда).

### Практика: наш `ci.yml` (3 job)
1. **hasher-tests:** `setup-python` → `pip install` → `pytest`.
2. **cpp-tests:** `apt install cmake g++` → `cmake -S tests` → `ctest`.
3. **integration:** создать фиктивный `.env` → `docker compose up --build` →
   дождаться `healthy` → smoke `curl` → `down`.

Условия: `if: failure()` — показать логи при падении; `if: always()` —
гарантированно прибрать стек.

### Вероятные вопросы
- *Что такое CI/CD?* — CI: авто-сборка и тесты на каждый коммит; CD:
  авто-доставка/деплой.
- *Что делает `actions/checkout`?* — Клонирует репозиторий в раннер.
- *Чем job отличается от step?* — Job — изолированная задача на своей ВМ; step
  — шаг внутри job.
- *Когда запускается workflow?* — По триггерам из `on` (push/PR в `main`).
- *Где смотреть результат?* — Вкладка **Actions**; галочка/крест у коммита.
